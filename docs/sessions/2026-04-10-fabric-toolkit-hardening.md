## Session Summary

Third pre-kickoff working session on t3-hearst. The goal going in was narrow — verify that the `.claude/` symlinks from last session actually let Claude Code load the fabric agents and skills, and exercise one of the fabric agents on a real Phase-2-adjacent design question. The session started with a clean structural check (all 7 symlinks resolve, agent/skill rosters enumerate correctly) and then spawned `FabricDataEngineer` on a WideOrbit medallion design straw-man. The agent produced strong output but flagged a structural gap in its own response: **skills are not auto-loaded into spawned subagents**. That single discovery redirected the rest of the session.

Root-caused the gap (Claude Code registers skills at main-session level; subagent contexts are isolated and do not inherit the parent's skill roster; the `delegates_to:` frontmatter on upstream fabric agents is advisory metadata from GitHub Copilot CLI's delegation model and is not walked by Claude Code's loader). Designed three workarounds, picked the subagent-reads-SKILL.md-directly option as the canonical pattern, then wrote a proper tagged plan (`fabric-toolkit-integration-hardening`) to address the gap in three places consistently: project `CLAUDE.md` as the entry-point signal for all spawned agents, `MANIFEST.md` as the agent-facing routing index with a "For agents" preamble, and `docs/fabric-toolkit/README.md` with a full Invocation Patterns section covering all three workaround options and an example spawn prompt. Executed the plan inline (6 tasks, 3 waves), with the middle wave being a **controlled comparison spawn** — the same `FabricDataEngineer` agent prompted on a narrower skill-shaped design task (Bronze→Silver SCD2 merge for WideOrbit spots) **with** the MANIFEST-first + SKILL.md-read instructions in the prompt. Compared that output head-to-head against the blind-baseline straw-man from earlier in the session.

**The validation result was decisive.** The test spawn actually Read 4 files (MANIFEST + 2 SKILL.md files + 1 resource doc — all declared in a "Files Read" header the agent produced on request), and the output was dramatically better than the baseline: full PySpark MERGE pseudocode where the baseline had hand-waving, concrete business-key derivation via hash functions (`sha1` / `xxhash64` / `sha2`) where the baseline was generic, two-pass SCD2 expire-then-insert with explicit reasoning for why a single MERGE doesn't work atomically, within-batch deduplication via window functions (addressing the WideOrbit-specific "same spot re-emitted in one export" problem the baseline didn't anticipate), 5 pre-merge + 3 post-merge quality gates tiered by severity, a refined single-table quarantine design that explicitly **contradicted** the baseline's per-entity quarantine recommendation with reasoning traceable to skill content, and liquid clustering as the primary recommendation over Z-ORDER. Both design refinements were explicitly attributed to skill references. The pattern is now the canonical fabric-agent invocation pattern in this repo, documented in three places, and proven to not just "work mechanically" but to **materially improve output quality**.

After closing the hardening plan, the session produced two pieces of downstream work based on explicit user asks. First: saved the skill-loaded test spawn output to `docs/plans/2026-04-10-wideorbit-spots-silver-merge-skill-loaded.md` as a durable artifact so the straw-man and the skill-loaded doc can sit side-by-side in `docs/plans/` as a visible proof point for the invocation-pattern framework. Second: recognized that the fabric toolkit is instance #1 of a much more general **Skill Locker pattern** — content store + curation index + invocation protocol + entry-point signal + optional specialized agents — and wrote a portable, instance-agnostic `SKILL-LOCKER-PATTERN.md` at `docs/` (189 lines, self-contained, no fabric or t3-hearst references). Surfaced identical copies of that pattern doc in 7 of Ken's non-sibling freelance projects (`KGiQ-LLC`, `portfolio-front-end`, `booking-module`, `open-mind`, `iq-publish`, `artemis-intel`, `upwork`) so the pattern is immediately accessible anywhere a new knowledge-locker candidate emerges. Explicitly parked the "should the locker itself live globally?" exploration for another session, but captured the framing of that question inside every copy of the pattern doc so it travels with the artifact.

Session ended at a natural stopping point: Pre-Kickoff Workstream 3 is marked Complete on the brief, all three pre-kickoff workstreams are now done, and the project is positioned to enter the formal SOW Plan phase at the Apr 13–15 kickoff with the fabric toolkit wired, verified, invocation-patterned, and framework-generalized. ## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Treat "skills don't auto-load into subagents" as a structural fact of Claude Code, not a bug to work around via framework changes | Verified the behavior (agent flagged it, then replicated it). The `delegates_to:` frontmatter on upstream fabric agents is advisory metadata designed for GitHub Copilot CLI's delegation model — Claude Code doesn't walk that list. Trying to patch Claude Code's loader is out of scope and fragile. File-based delivery via Read is the clean escape hatch. |
| Adopt **subagent Reads SKILL.md directly** as the canonical default pattern, not "inline skill content in spawn prompt" | Cheapest, most repeatable, preserves subtree-sync purity (zero upstream edits required), works for any future fabric skill without per-task work by the calling thread. The agent also gets to judge *which* skills to load based on task context — judgment belongs at the agent level, not pre-computed by the spawner. Options #2 and #3 remain in the documented toolkit for cases where the default isn't ideal. |
| Document the pattern in **three places** (CLAUDE.md, MANIFEST.md, fabric-toolkit README.md) rather than a single canonical location | Each location serves a different audience: CLAUDE.md is read by every spawned agent regardless of task (entry-point signal); MANIFEST.md is read by agents doing fabric work (routing index); fabric-toolkit README.md is read by humans onboarding or debugging the toolkit (reasoning + alternatives + examples). Single-source-of-truth would save ~20 lines of duplication at the cost of each audience needing to chase references — not worth it. |
| Extend the brief with a new **Pre-Kickoff Workstreams table** rather than folding pre-kickoff tooling into the SOW Phase Tracker | The SOW Phase Tracker is sacred structure — it reflects the contract (Plan, Discovery, Design, Execute, Closure). ## Discoveries

- **Discovery:** Claude Code skills are registered at the **main session level only**. Subagents spawned via the Agent tool get an isolated context that does not inherit the parent session's skill roster. The fabric agents' upstream `delegates_to:` frontmatter is advisory metadata (designed for GitHub Copilot CLI's delegation model) and is not walked by Claude Code's loader to pre-register listed skills into the subagent.
  - **Impact:** This is the single most important wrinkle for how `/om-exec` and any wave-based execution use fabric agents. If a wave spawns `FabricDataEngineer` to build a Bronze→Silver notebook, the agent falls back to embedded knowledge and does not benefit from the detailed skill content unless that content is surfaced another way. **Downstream:** All fabric-invocation patterns in this repo now adopt the "subagent Reads SKILL.md directly" workaround as the canonical default. If Claude Code ever changes this behavior and starts auto-loading skills into subagents, the workaround becomes unnecessary but the locker pattern still holds value (curation, filtering, progressive disclosure, trust-boundary separation).

- **Discovery:** The workaround doesn't just "work mechanically" — it **materially improves output quality**. The T4 controlled-comparison spawn demonstrated that a skill-loaded `FabricDataEngineer` produces output that is both deeper and better-grounded than a blind-baseline spawn on comparable scope: full PySpark MERGE pseudocode (baseline had none), concrete hash-function key derivation (baseline was generic), within-batch deduplication handling the WideOrbit "same spot re-emitted" case (baseline missed this entirely), tiered quality gates with severity-driven routing (baseline named gates abstractly), and **two explicit design refinements** over the straw-man (single-table quarantine with gate-level tagging, liquid clustering over Z-ORDER) both traceable to specific skill references.
  - **Impact:** Elevates the MANIFEST-first + SKILL.md-read pattern from "useful convention" to "demonstrably load-bearing framework." **Downstream:** This is the graduation moment where the invocation pattern stops being a workaround and starts being the primary way to use fabric agents. It should be the default treatment for any spawn touching fabric-adjacent work.

- **Discovery:** The fabric toolkit is instance #1 of a more general **Skill Locker pattern** with five parts: content store, curation index, invocation protocol, entry-point signal, and optional specialized agents. The shape has no fabric-specific assumptions baked in — the same scaffold works for any dense, progressively-disclosed knowledge domain (other cloud platforms, industry/domain knowledge, internal codebase knowledge like the Stripe agentic-coding angle, team delivery patterns, project-local historical context).
  - **Impact:** The work already done on the fabric toolkit is transferable. Any future knowledge domain in any of Ken's projects that matches the "dense + stable + reused" criteria can instantiate the same pattern without re-deriving the architecture. **Downstream:** The portable pattern doc has been placed in 7 freelance projects outside t3-hearst. When a second real locker candidate emerges in any of those contexts, there's a working template to reach for. When two projects instantiate the same kind of locker (e.g., both want a "web framework conventions" store), that's the trigger to revisit the parked global-hosting question.

- **Discovery:** The Claude Flow daemon is dying **reproducibly** during sessions, not as a one-off. This is the second consecutive session with the failure, and within this session alone the daemon died three separate times (at session start before `/om-go`, again during working hours before `/om-save`, and again before `/om-wrap`). Last session's hypothesis correlated the death with `git subtree` operations; this session had no subtree operations but still died repeatedly, which weakens that hypothesis. The failure is silent — no error, just the PID process disappearing. Restart from PROJECT_ROOT brings it back cleanly each time.
  - **Impact:** Every session that depends on Claude Flow memory needs a daemon-guard check + auto-restart fallback, which is already present in `/om-go`, `/om-save`, and `/om-wrap`. But the failure is repeated and worth a dedicated investigation. **Downstream:** Next session should include a planned experiment — leave the daemon running in a monitored state and identify what activity (if any) correlates with the process disappearance. Candidates to rule out: specific MCP calls, heavy Agent tool spawning, specific file write patterns, memory pressure.

- **Discovery:** The upstream fabric agents (`FabricDataEngineer`, `FabricAdmin`) are listed in Claude Code's agent roster with **PascalCase names** from their YAML frontmatter, not with kebab-case from the symlink filenames (`fabric-data-engineer.md`, `fabric-admin.md`). Claude Code reads `name:` from frontmatter, not the filename.
  - **Impact:** When spawning via the `subagent_type` parameter of the Agent tool, use `FabricDataEngineer` / `FabricAdmin` (the frontmatter `name`), not the filename slug. A future wave-executor or router that keys off kebab-case conventions will need a name-alias map or casing-normalization shim. **Downstream:** Cosmetic issue today, but worth remembering when extending `/om-plan` and `/om-exec` vocabulary to natively recognize fabric agents — they'll need to handle both cases.
